# Proactive Layer Report

## Purpose
The proactive layer is the part of the product that surfaces merchant-relevant signals before the merchant explicitly asks for them.

In the current system, it is responsible for:
- generating background cards
- ranking and persisting those cards
- preserving card state across refreshes
- optionally converting a card into an Action Center item
- exposing those signals to the UI as a merchant inbox

## Current Architecture
There are two proactive signal sources:

1. Template-driven insight cards
- generated by `generate_insight_cards(...)`
- based on a fixed catalog of card templates under [`/Users/madhavpatel/New_demo copy/agent/CARDS`](/Users/madhavpatel/New_demo copy/agent/CARDS)
- powered by KPI-like metrics built from merchant data

2. Deterministic payout shortfall alerts
- generated by `generate_payout_shortfall_alerts(...)`
- based on actual settlement gross-vs-net gaps
- each alert also produces an action payload

These are merged in `refresh_background_proactive_cards(...)`, ranked, persisted to `proactive_cards`, and later shown in the inbox.

## Core Logic
### Template insight cards
`generate_insight_cards(...)`:
- loads all card templates
- builds merchant metrics
- evaluates each template condition
- renders title/body/actions from metrics
- returns a list of card payloads

Strength:
- simple and explainable
- deterministic

Weakness:
- limited to the current catalog
- coverage depends entirely on prewritten templates and thresholds

### Payout shortfall alerts
`generate_payout_shortfall_alerts(...)`:
- scans `settlements` for rows where `gross_amount - net_settlement_amount >= min_difference`
- orders by largest gap
- runs the deterministic shortfall explainer on each candidate
- produces both:
  - a card payload
  - a matching action payload

Strength:
- high-trust and demo-relevant
- grounded in real settlement math

Weakness:
- settlement-centric, not merchant-summary-centric
- currently defaults to multiple settlement-level alerts per merchant/window

### Persistence and ranking
`refresh_background_proactive_cards(...)`:
- determines the current window
- generates template cards and shortfall cards
- ranks by `impact_rupees` then `confidence`
- deletes same-window background cards
- reinserts fresh cards while preserving:
  - `card_state`
  - `card_notes`
  - `converted_action_id`
- auto-creates an Action Center item for shortfall alerts

Strength:
- refresh is deterministic and window-scoped
- card state survives refreshes

Weakness:
- same-window delete/reinsert is simple, but it is not event-native
- template cards and shortfall cards are merged late rather than sharing a canonical event model

## Current Card State Model
The proactive layer supports card states:
- `NEW`
- `ACKNOWLEDGED`
- `DISMISSED`
- `CONVERTED`

It also stores:
- notes
- linked/converted action id
- window
- evidence ids

This is a useful operational foundation.

## UI Behavior
The React inbox lets the user:
- refresh cards
- acknowledge a card
- dismiss a card
- add notes
- preview action payload
- create an action from a card

Strength:
- gives the proactive layer a real workflow surface

Weakness:
- the UI is still just a consumer; it does not improve the signal quality itself
- card presentation is currently one-card-per-signal, with no merchant-summary grouping

## Strengths of the Current Proactive Layer
1. It is deterministic and understandable.
2. It supports persistence, dedupe, state, and notes.
3. It now includes a strong payout-shortfall signal type.
4. It can convert signals into actions.
5. It is UI-agnostic enough to survive future frontend changes.

## Main Weaknesses
1. Trigger coverage is narrow.
- most merchants will not light up many cards
- several operational scenarios are still missing

2. Thresholds are not merchant-segment-aware.
- large and small merchants are judged through broadly similar trigger logic

3. Shortfall alerts are too granular for presentation.
- one merchant can receive multiple settlement-specific cards in the same window
- good for drilldown, weak for top-level UX

4. There are still two proactive generation concepts in the product.
- background proactive cards in `merchant_os.py`
- runtime-side proactive persistence in the chat runtime
- they overlap conceptually without a single canonical event pipeline

5. The system is still warning-heavy.
- limited “positive opportunity” or “light guidance” coverage for quieter merchants

## Immediate Refinement Recommendations
1. Introduce a canonical proactive event model.
- event type
- severity
- evidence
- recommended action
- presentation strategy

2. Aggregate merchant-level operational summaries.
- especially for payout shortfalls
- preferred format for demo and product UX:
  - one summary event
  - optional drilldown rows

3. Expand operations trigger coverage.
- overdue chargebacks
- unresolved payout mismatches
- processed settlement shortfalls
- unresolved refund exceptions

4. Add merchant-segment-aware thresholds.
- low-volume merchants should still get meaningful signals
- high-volume merchants should avoid noisy alerts

5. Clarify ownership of proactive generation.
- choose one canonical proactive pipeline
- keep the other as an adapter/consumer

## Overall Assessment
The proactive layer is already functional enough to support the internal demo story, especially after payout shortfall integration.

It is not yet broad or clean enough to represent a mature merchant-wide monitoring system.

### Readiness
- Internal demo readiness: strong enough with summary hardening
- Pilot readiness: partial
- Long-term merchant OS readiness: not yet; trigger breadth and event-model cleanup are still needed
